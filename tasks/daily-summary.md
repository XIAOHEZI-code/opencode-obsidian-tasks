# 任务：生成每日复盘并自动填充全部笔记字段

本任务不再局限于仅写入「每日复盘」章节。在扫描当日活动后，应自动填充每日笔记的 **frontmatter 元数据**（tags、score）和 **内容区块**（📋 今日任务、📝 笔记与想法），最后再追加每日复盘。目标是让 AI 完成整篇每日笔记的骨架，用户仅需微调。

## ⚠️ 代理委派策略

本任务的 build agent 需要将具体工作委托给子代理。**务必按以下规则分配**：

| 工作类型 | 使用子代理 | 原因 |
| :--: | :--: | :--: |
| 扫描 git 仓库 / 本地文件 | `explore` | explore 具备完整的 find/ls/git log 权限 |
| 外部资料查询 | `researcher` | researcher 可访问 GitHub/npm 等外部源 |
| 兜底 | build agent 自己跑 bash | 如果子代理全部失败，不要放弃——build agent 已被授予 find/ls/git 权限，可直接执行命令 |

> ⚡ 关键：子代理失败一次不代表任务失败。先换策略，再降级到直接执行，最后才放弃。

## 📂 参数配置

| 参数 | 值 | 说明 |
| :--: | :--: | :--: |
| Git 扫描根目录 | `~/projects/` | 递归扫描所有子目录中的 git 仓库 |
| Obsidian Vault | `<YOUR_VAULT_PATH>/` | 每日笔记在 vault 根目录 |
| 看板笔记 | `2-技能/定时任务看板.md` | 执行日志写入目标 |
| 参考示例 | `2026-06-17.md` | 完整每日笔记格式参考（含 tags/score/tasks/notes/review） |

> 💡 迁移到其他机器时，只需修改上表中的路径。

## ⏰ 日期边界规则（6:00 AM 分界）

每日笔记的复盘目标日期**不是简单的"今天"**。以凌晨 6:00 为日切点：

- 如果当前时间 **< 6:00 AM**，则复盘目标是**昨天**，写入昨天的每日笔记文件
- 如果当前时间 **≥ 6:00 AM**，则复盘目标是**今天**，写入今天的每日笔记文件

**文件命名**：每日笔记文件名必须匹配复盘目标日期，格式为 `YYYY-MM-DD.md`（例如 `2026-06-18.md`）。

> 📌 示例：当前时间 2026-06-19 00:13 → 复盘目标为 2026-06-18 → 写入文件 `2026-06-18.md`

在执行任何扫描之前，先通过 `date` 命令确定当前时间，计算出正确的复盘目标日期。

## 执行步骤

### 1. 扫描 Git 提交
扫描 `~/projects/` 下所有 git 仓库在复盘目标日期的提交记录：
- 对于每个仓库，运行 `git log --since="YYYY-MM-DD" --until="YYYY-MM-DD+1" --oneline --all`（日期替换为复盘目标日期）
- 记录项目名、commit hash、提交信息、日期

### 2. 扫描 Obsidian 笔记变更
- 使用 Obsidian 工具检查 vault 中复盘目标日期创建或修改的笔记
- 按分类整理（系统运维 / 项目开发 / 技术研究 / 其他）

### 3. 生成复盘内容（每日复盘章节）
将以上信息组织成「每日复盘」章节格式。此章节稍后与其他自动生成的内容一起写入每日笔记：

```markdown
## 每日复盘

### 📊 活动汇总 (YYYY-MM-DD)

#### 💻 Git 提交
- **项目名**
  - `commit_hash` - 提交信息 (日期 时间)
  - ...

#### 📝 Obsidian 笔记变更
- **分类名**：笔记1、笔记2...

---

### 💡 AI 提升建议

1. **建议标题**
   - **观察**：...
   - **建议**：...
```

### 3.2. 自动填充 Frontmatter 字段

在生成复盘内容后，根据扫描到的当日活动，自动分析并填充每日笔记的 frontmatter 元数据。

#### 3.2.1 自动填充 Tags

分析当日活动，提炼出 **5-8 个标签**（含 `daily`），其中 tag1-tag3 为核心主题标签。

**操作步骤**：
1. 审视当日的 git 提交、笔记变更、项目产出，归纳出 3-5 个主题关键词
2. 组成标签列表：`["daily", "主题1", "主题2", "主题3", ...]`
3. 使用 `obsidian_patch_note` 更新 frontmatter 的 tags 字段：
   ```
   obsidian_patch_note(path="YYYY-MM-DD.md", operation="replace", target_type="frontmatter", target="tags", content='["daily", "安全", "逆向测试", "Web安全", ...]')
   ```
4. 在笔记正文顶部（frontmatter 之后、标题之前）追加一行内联标签：
   ```
   #主题1 #主题2 #主题3
   ```

#### 3.2.2 自动填充评分

基于当日活动的数量和质量，估算 5 个维度的分数，并更新 frontmatter 的 `score` 字段。

**评分维度与标准**：

| 维度 | 说明 | 评分依据 |
| :--: | :--: | :--: |
| 任务密度 | 完成事项的数量和产出密度 | 提交数 + 笔记数 + 交付文档数。0-100 |
| 深度系数 | 技术工作的复杂度 | 是否涉及源码分析、逆向、架构设计等。0-100 |
| 闭环度 | 任务从启动到完成的比例 | 已闭环事项 / 总事项。0-100 |
| 复利指数 | 产出对未来工作的可复用程度 | 是否产出模板、SOP、工具、方法论。0-100 |
| 跨界宽度 | 涉及领域的多样性 | 同日触及的技术域/业务域数量。0-100 |

- **total** = 上述 5 项的平均值（四舍五入）
- **level** = 根据 total 映射：A(≥80) / B(65-79) / C(50-64) / D(35-49) / E(<35)

**操作步骤**：
1. 根据扫描结果，为每个维度打分并记录简短理由（一句话）
2. 使用 `obsidian_patch_note` 更新 frontmatter 的 `score` 字段：
   ```
   obsidian_patch_note(path="YYYY-MM-DD.md", operation="replace", target_type="frontmatter", target="score", content='{"density": 85, "depth": 90, "closure": 80, "compound": 85, "breadth": 75, "total": 83, "level": "A"}')
   ```
3. **mood 字段留空**（用户自行填写）

### 3.5. 检查今日笔记质量

在自动填充完成后，读取每日笔记验证所有字段是否已正确填充：

| 检查项 | 判断标准 | 不达标时 |
| :--: | :--: | :--: |
| Tags 丰富度 | frontmatter tags 包含 `daily` 及至少 3 个主题标签 | 追加标签 |
| 内部链接数 | wiki-links ≥ 3 个 | 在复盘建议中追加「📋 建议增加内部链接」 |
| 评分是否填写 | score.total > 0 | 重新估算并补填 |
| 今日任务是否生成 | 📋 今日任务 区块存在且 ≥ 3 项 | 重新生成 |
| 笔记与想法是否生成 | 📝 笔记与想法 区块存在且 ≥ 100 字 | 重新生成 |

将检查结果以「📋 笔记健康提示」格式嵌入复盘内容的 AI 提升建议区块中（放在第 1 条建议之前），格式如下：

```
### 📋 笔记健康提示
- ✅ Tags 已自动填充：`#安全 #逆向测试 #Web安全`
- ⚠️ 内部链接过少（当前仅 X 个），建议链接相关笔记
- ✅ 评分已自动估算：83/A
```

### 3.7. 自动填充内容区块

在质量检查通过后，自动生成每日笔记的两个核心内容区块。

#### 3.7.1 📋 今日任务

根据当日的 git 提交、笔记变更、项目产出，生成 3-8 条任务项。已完成的事项标记 `[x]`，进行中的标记 `[ ]`。

**格式**：
```markdown
## 📋 今日任务
- [x] 完成转炉炼钢逆向测试报告撰写（11 篇文档交付）
- [x] 提炼 Web 安全认证绕过技术体系 → [[Web安全认证绕过技术全览]]
- [ ] 安全学习路线实战验证（ATT&CK ID 标注到漏洞报告）
```

> 任务项应具体、可验证。优先使用 wiki-links 引用相关笔记。

#### 3.7.2 📝 笔记与想法

撰写一段 200-400 字的当日工作摘要，使用 wiki-links 引用关键笔记。如有值得提炼的洞察，以 blockquote 形式附加。

**格式**：
```markdown
## 📝 笔记与想法

### 🧠 知识沉淀：{主题}

今日核心工作摘要（200-400 字），概述产出、关联笔记、关键发现。

| 关键发现 | 值 |
| :--: | :--: |
| 发现1 | 值1 |
| 发现2 | 值2 |

> 💡 当日最重要的一个洞察或收获。
```

#### 3.7.3 写入内容区块

使用 `obsidian_patch_note` 将生成的内容区块写入每日笔记：
- 📋 今日任务 → 插入到 `## 每日复盘` 标题之前
- 📝 笔记与想法 → 插入到 `## 📋 今日任务` 之后、`## 🏆 今日评分` 之前

**写入顺序**：先插入 📋 今日任务，再插入 📝 笔记与想法。

### 4. 写入 Obsidian（含降级）

整合所有已生成的内容（📋 今日任务 + 📝 笔记与想法 + 🏆 今日评分 + 每日复盘），优先使用 Obsidian 工具写入每日笔记。

**优先路径**（Obsidian API 可用时）：
- 使用 `obsidian_patch_note` 逐区块追加，保留已有内容不覆盖
- frontmatter 字段已通过 Step 3.2 用 `obsidian_patch_note` 更新

**降级路径**（Obsidian API 不可用时）：
- 使用 `read` 检查 `<YOUR_VAULT_PATH>/YYYY-MM-DD.md` 是否已存在
- 如果不存在，使用 `write` 创建文件并写入完整的 frontmatter + 内容区块
- 如果已存在，先 `read` 读取已有内容，智能合并（已有区块不覆盖、缺失区块追加），再 `write` 写入
- 确保不覆盖已有内容，始终保留原有笔记数据

### 5. 追加执行记录到看板（含失败记录）

无论任务成功或失败，都必须在 Obsidian vault 的 `2-技能/定时任务看板.md` 中追加一条执行记录。

**成功时**（全部字段和内容区块均已写入）：
```
- ⏱️ YYYY-MM-DD HH:MM | ✅ daily-summary | ⏱️ ~Xs | 发现 X 提交, X 笔记变更 | tags/score/tasks/notes/review 已自动填充
```

**部分成功时**（部分写入失败）：
```
- ⏱️ YYYY-MM-DD HH:MM | ⚠️ daily-summary | ⏱️ ~Xs | 扫描成功但写入失败：{原因} | 已完成: {已完成的字段}
```

**完全失败时**（无法完成扫描）：
```
- ⏱️ YYYY-MM-DD HH:MM | ❌ daily-summary | ⏱️ ~Xs | 失败：{具体原因}
```

操作方式：
- 使用 `obsidian_append_to_note`（优先）或 `read` + `write`（降级）追加到看板笔记末尾
- 不要覆盖看板已有内容
- 执行记录是唯一的任务可观测性出口——**必须写入**，即使前面全部失败

## 参考格式
参考 Obsidian vault 中的 `2026-06-17.md` 笔记——它展示了完整的每日笔记格式，包含：
- frontmatter（tags / score / mood）
- 内联标签行
- 📋 今日任务（checkbox 列表）
- 📝 笔记与想法（摘要 + blockquote 洞察）
- 🏆 今日评分（5 维评分表）
- 📊 周回顾（dataview 查询）
- 每日复盘（活动汇总 + AI 提升建议）

所有自动生成的内容应与此格式对齐。

## 注意事项
- **日期边界优先**：在任何操作之前，先按 6:00 AM 规则确定复盘目标日期
- 如果复盘目标日期没有任何 git 提交或笔记变更，如实说明并仍填充基本字段
- 复盘内容要简洁、有条理
- AI 建议要具体、可操作，每条建议包含"观察 → 建议"结构
- 如果每日笔记还不存在，按完整格式创建
- frontmatter 更新使用 `obsidian_patch_note`（优先）或 `write`（降级）
- 所有自动生成的内容应标注生成来源（如 `*🤖 由 AI 每日复盘任务生成 · YYYY-MM-DD HH:MM*`）
- 降级文件写入时，确保不覆盖已有内容；先 `read` 读取已有内容，合并后再 `write`
- mood 字段始终留空，由用户手动填写
- 执行记录是唯一可观测性出口，必须写入看板
